# SW - Library / API / Framework / Engine / SDK

## 개요
- **핵심 개념**: 소프트웨어 재사용 단위의 종류와 제어 흐름의 차이
- **핵심 구분 포인트**: **"누가 누구를 호출하는가"**

## 상세 분석

### 한눈에 비교

| 개념 | 제어 주체 | 범위 | 예시 |
|------|----------|------|------|
| Library | 내 코드 → Library 호출 | 함수/모듈 단위 | OpenCV, Eigen |
| API | 내 코드 → 외부 시스템 호출 | 인터페이스 정의 | REST API, OS API |
| Framework | Framework → 내 코드 호출 | 애플리케이션 구조 전체 | ROS2, Django |
| Engine | Engine이 핵심 기능 수행 | 특정 목적의 핵심 처리 | 게임 엔진, 검색 엔진 |
| SDK | 개발 도구 + API + Library 묶음 | 통합 개발 환경 | Android SDK, ROS2 SDK |

### Library (라이브러리)

- 자주 사용되는 기능을 모아놓은 코드 묶음
- **내 코드가 라이브러리를 호출** (제어 흐름: 내 코드 주도)

#### 정적 라이브러리 (Static Library)

| | 내용 |
|-|------|
| 방식 | 컴파일 시점에 실행 파일에 포함 |
| 장점 | 시스템 환경 변화에 영향 없음, 배포 간단 |
| 단점 | 실행 파일 크기 증가, 메모리 중복 로드 가능 |

#### 동적 라이브러리 (Dynamic Library)

| 종류 | 내용 |
|------|------|
| 동적 링크 | 실행 파일 로드 시 라이브러리 함께 메모리에 로드 |
| 동적 로드 | 런타임 중 필요할 때 `dlopen()` 등으로 명시적 로드 |

- 장점: 빌드 시점에 없던 라이브러리도 사용 가능 (플러그인 시스템)
- 단점: 경로 설정 필요, 버전 의존성 관리 필요

### API (Application Programming Interface)

- OS나 외부 시스템이 제공하는 기능을 사용하기 위한 **인터페이스 규약**
- 구현 내용은 숨기고 사용 방법(함수 시그니처, 엔드포인트 등)만 공개

### Framework

- 애플리케이션의 **구조(뼈대)와 흐름을 결정**하는 소프트웨어
- **Framework가 내 코드를 호출** (제어 역전, IoC — Inversion of Control)
- 특징: 설계자가 의도한 디자인 패턴 집합 + 확장 가능한 기반 코드

> "Library를 쓸 때는 내가 코드를 쓴다. Framework를 쓸 때는 Framework가 내 코드를 쓴다."

### Engine

- 특정 목적을 위한 **핵심 처리** 담당 프로그램
- 예: 검색 엔진(인덱싱·쿼리 처리), 게임 엔진(렌더링·물리 처리)

### SDK (Software Development Kit)

- 특정 플랫폼/환경에서 개발하기 위한 **도구 묶음** (API + Library + 샘플 + 문서)
- 현재는 API와 거의 동의어로 사용되는 경우도 많음
